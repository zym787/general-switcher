# general-switcher 重构计划

## 1. 背景与核心问题

当前项目是 STM32/Keil C 工程，主要业务集中在 `SRC/APP`、`SRC/HARDWARE` 和 `SRC/SYSTEM`。代码规模不算特别大，但耦合程度较高，主要问题包括：

- `common.h` 成为巨型公共头文件，包含几乎所有模块，导致依赖方向混乱。
- `main.h` 同时承载版本信息、EEPROM 地址、类型定义、全局变量声明和函数声明。
- `main.c` 同时负责启动流程、参数初始化、业务调度、超时保护、协议选择和 LED 错误提示。
- `modbus.c`、`ags_mb.c`、`motor.c`、`usFunc.c` 等模块直接读写 `syspara`、`valve`、`valveFix`、`rdc`、`modbus` 等全局变量。
- 协议层直接知道业务状态和 EEPROM 细节，导致协议解析、寄存器映射、业务动作混在一起。
- 阀控制状态使用 bit flag 组合表达，状态跳转入口分散，后续维护风险高。
- 多硬件版本依赖大量条件编译，版本差异与业务逻辑交织。

重构原则：

- 不一开始消灭所有全局变量，先集中，再封装，最后逐步传上下文。
- 不改变 EEPROM 地址、协议寄存器地址、默认参数值、CRC 顺序、启动时序。
- 每个阶段只解决一类问题，避免参数、协议、状态机、硬件抽象混在一个提交中。
- 每次重构后至少保证关键 Keil Target 可编译。

## 2. 当前结构

当前结构可以概括为：

```text
common.h
  |-- include 几乎所有头文件
  |-- 定义硬件版本宏
  |-- 定义 GPIO 宏
  |-- 定义调试宏
  |-- 引入第三方日志和工具库

main.h
  |-- 软件版本
  |-- EEPROM 地址
  |-- 协议 enum
  |-- 系统参数 struct
  |-- syspara 全局变量

main.c
  |-- 硬件初始化
  |-- 参数读取/默认写入
  |-- 协议初始化
  |-- 阀控制调度
  |-- 超时保护
  |-- LED 错误提示

modbus.c / ags_mb.c
  |-- 协议帧处理
  |-- 寄存器读写
  |-- 直接修改业务全局变量

motor.c / motor.h
  |-- 阀状态
  |-- GPIO 映射
  |-- 速度、减速比、通道数等业务常量
```

目标结构：

```text
SRC/
  APP/
    app_main.c              应用初始化和主循环编排
    app_context.c/.h        应用上下文，集中业务状态
    app_param.c/.h          参数加载、校验、默认值、持久化
    app_param_map.h         EEPROM 地址和长度定义
    app_service.c/.h        业务服务接口，供协议层调用
    app_version.c/.h        版本信息

  BOARD/
    board_config.h          硬件型号、编译 Target 配置
    board_pins.h            GPIO/引脚映射
    board_io.c/.h           LED、RS485、阀控制引脚封装

  VALVE/
    valve.c/.h              阀状态机和阀动作
    valve_types.h           阀状态、事件、配置类型
    valve_config.c/.h       减速比、步数、速度范围

  PROTOCOL/
    ags_protocol.c/.h       AGS 协议帧处理
    modbus_protocol.c/.h    Modbus 协议帧处理
    register_map.c/.h       寄存器映射和读写分发

  DRIVER/
    EEPROM、USART、TIMER、FLASH 等底层驱动

  3rd/
    第三方库
```

注意：目录重组不要第一阶段就做。Keil 工程依赖路径较多，应先新增文件和薄封装，稳定后再移动目录。

## 3. 编译与验证基线

### 3.1 当前 Keil Target

`USER/QHF.uvprojx` 中包含以下 Target：

```text
ValveX2
O_901
O_906
O_909
A_901
A_906
A_909
B_901
B_906
C_901
```

### 3.2 每阶段验证要求

每个阶段完成后至少检查：

- 主 Target 可编译。
- 至少一个 901、一个 906、一个 909 Target 可编译。
- AGS 协议和 Modbus 协议入口仍能按 `syspara.protocol_type` 初始化。
- EEPROM 参数地址和长度不变。
- 默认参数值不变。
- 主循环调用顺序不发生非预期改变。

建议完整阶段结束时编译全部 Target。

## 4. 第一阶段：拆分公共头文件

目标：降低头文件耦合，但不改变业务行为。

### 4.1 新增 `base_types.h`

职责：

- 定义基础类型。
- 引入 `stdint.h` 和 `stdbool.h`。
- 作为 `uint8/uint16/uint32` 到标准类型迁移的过渡层。

建议内容：

```c
#ifndef BASE_TYPES_H
#define BASE_TYPES_H

#include <stdbool.h>
#include <stdint.h>

typedef unsigned char  uint8;
typedef unsigned short uint16;
typedef unsigned int   uint32;

#endif
```

### 4.2 新增 `board_config.h`

职责：

- 管理硬件型号和编译 Target 选择。
- 从 `common.h` 迁出 `O_901`、`A_906`、`B_906`、`C_901` 等条件编译逻辑。
- 统一产生 `A12_901`、`A12_906`、`A12_909`、`IOCTRL`、`IO_RS`、`RS232_485_CONTROL`、`DIRECTION_SWITCH` 等宏。

迁移内容：

- 版本选择宏。
- IO 极性宏。
- `PCB_VR`。
- `HARDWARE_DESCRIPTION`。

验收标准：

- `common.h` 不再直接承载硬件 Target 推导逻辑。
- 所有原有 Target 宏展开结果保持一致。

### 4.3 新增 `debug_config.h`

职责：

- 管理 `RELEASE`、`DEBUG`、`DEBUG_MODBUS`。
- 定义 `dbg_printf`。

验收标准：

- 调试开关逻辑与现有行为一致。
- `RELEASE` 模式下调试输出仍被关闭。

### 4.4 新增 `board_pins.h`

职责：

- 暂时集中 GPIO 宏映射。
- 从 `motor.h` 和 `common.h` 中迁出硬件引脚定义。

迁移内容：

- `LED_WORK`
- `VALVE_OPT`
- `VALVE_ENA`
- `VALVE_RST`
- `VALVE_DIR`
- `VALVE_CLK`
- `M_ISET_1`
- `M_ISET_2`
- `M_ISET_3`
- `RX_EN()`
- `TX_EN()`

后续这些宏会被 `board_io.c/.h` 的函数封装替代。

### 4.5 保留过渡头

短期内可以保留 `common.h`，但它应逐步变成：

```c
#ifndef _COMMON_H_
#define _COMMON_H_

#include "base_types.h"
#include "board_config.h"
#include "debug_config.h"
#include "board_pins.h"

/* 临时兼容 include，后续逐步删除 */

#endif
```

## 5. 第二阶段：集中应用状态

目标：先把散落全局变量集中到一个上下文中，再逐步隐藏字段。

### 5.1 新增 `app_context.h`

建议定义：

```c
#ifndef APP_CONTEXT_H
#define APP_CONTEXT_H

#include "main.h"
#include "motor.h"

typedef struct {
    _SYS_T sys;
    _VALVE_T valve;
    _12VALVE_FIX valveFix;
    RDC_T rdc;
} AppContext;

extern AppContext g_app;

#endif
```

### 5.2 新增 `app_context.c`

```c
#include "app_context.h"

AppContext g_app;
```

### 5.3 迁移策略

第一步不要立即改所有模块函数签名。先把：

```c
syspara.xxx
valve.xxx
valveFix.xxx
rdc.xxx
```

逐步替换成：

```c
g_app.sys.xxx
g_app.valve.xxx
g_app.valveFix.xxx
g_app.rdc.xxx
```

后续再把核心函数改成：

```c
void AppParam_Load(AppContext *ctx);
void Valve_Process(AppContext *ctx);
AppResult App_SetSpeed(AppContext *ctx, uint8_t speed);
```

### 5.4 协议状态单独处理

建议协议层状态暂时保持独立，不混入业务上下文：

```c
extern Modbus_T modbus;
extern _AGS_MB_PRARM_T ags_mbParam;
```

后续可分别重构为：

```c
ModbusContext g_modbus;
AgsContext g_ags;
```

原因：

- 协议收发缓冲区生命周期与业务状态不同。
- Modbus/AGS 可以共享业务服务，但不应共享内部帧缓冲。

## 6. 第三阶段：重构参数系统

目标：把 EEPROM 参数读写、默认值、校验、修复从 `main.c` 中移出。

重点文件：

- `SRC/APP/main.c` 中的 `param_Read()`
- `SRC/APP/main.c` 中的 `param_Write()`
- `SRC/APP/main.c` 中的 `ParameterInit()`
- `SRC/APP/main.h` 中的 EEPROM 地址和长度定义

### 6.1 新增文件

```text
SRC/APP/app_param.h
SRC/APP/app_param.c
SRC/APP/app_param_map.h
SRC/APP/app_param_defaults.h
```

### 6.2 `app_param_map.h`

职责：

- 只定义 EEPROM 地址和长度。
- 从 `main.h` 迁出所有 `ADDR_*` 和 `LEN_*`。

示例：

```c
#define ADDR_BOARD_ID      0
#define LEN_BOARD_ID       2

#define ADDR_MODULE_NUM    (ADDR_BOARD_ID + LEN_BOARD_ID)
#define LEN_MODULE_NUM     1
```

验收标准：

- 地址计算结果与迁移前完全一致。
- 不在该文件中放业务默认值。

### 6.3 `app_param_defaults.h`

职责：

- 管理默认参数。

示例：

```c
#define PARAM_DEFAULT_ADDR              AGS_ADDR_DEF
#define PARAM_DEFAULT_VALVE_FIX_ORG     5
#define PARAM_DEFAULT_VALVE_FIX_DIR     0
#define PARAM_DEFAULT_RDC_RATE          RDC04
#define PARAM_DEFAULT_AGING_INTERVAL    5
#define PARAM_DEFAULT_ISET              0
```

### 6.4 `app_param.h`

建议接口：

```c
#ifndef APP_PARAM_H
#define APP_PARAM_H

#include "app_context.h"

void AppParam_Init(AppContext *ctx);
void AppParam_Load(AppContext *ctx);
void AppParam_SaveDefaults(AppContext *ctx);
void AppParam_RepairInvalid(AppContext *ctx);

#endif
```

### 6.5 `app_param.c`

建议先抽小函数，避免一次性表驱动过度重构：

```c
static uint8_t Param_ReadU8(uint16_t addr);
static void Param_WriteU8(uint16_t addr, uint8_t value);
static uint32_t Param_ReadU32(uint16_t addr);
static void Param_WriteU32(uint16_t addr, uint32_t value);

static bool Param_IsInRangeU8(uint8_t value, uint8_t min, uint8_t max);
static uint8_t Param_RepairU8(uint16_t addr, uint8_t value, uint8_t def, uint8_t min, uint8_t max);
```

### 6.6 拆分 `param_Read()`

建议拆成：

```c
static void AppParam_LoadAddress(AppContext *ctx);
static void AppParam_LoadValveFix(AppContext *ctx);
static void AppParam_LoadChannelCount(AppContext *ctx);
static void AppParam_LoadBaudrate(AppContext *ctx);
static void AppParam_LoadSpeed(AppContext *ctx);
static void AppParam_LoadProtocol(AppContext *ctx);
static void AppParam_LoadRdcRate(AppContext *ctx);
static void AppParam_LoadHalfSeal(AppContext *ctx);
static void AppParam_LoadCounters(AppContext *ctx);
```

### 6.7 拆分 `param_Write()`

建议拆成：

```c
static void AppParam_WriteBoardId(void);
static void AppParam_WriteDefaultAddress(AppContext *ctx);
static void AppParam_WriteDefaultValveFix(AppContext *ctx);
static void AppParam_WriteDefaultProtocol(AppContext *ctx);
static void AppParam_WriteDefaultRdc(AppContext *ctx);
static void AppParam_WriteDefaultReplyMode(AppContext *ctx);
```

### 6.8 参数阶段验收标准

- `main.c` 不再包含大段 EEPROM 参数读写逻辑。
- `ParameterInit()` 替换为 `AppParam_Init(&g_app)` 或等价调用。
- EEPROM 地址、长度、默认值保持不变。
- `modbus.Address` 与 AGS 地址初始化行为不变。
- `bsp_ValveUpdateSpeed(INIT_SPD)` 的调用时机不变。
- 初始化完成后 `valve.status` 和 `valve.ErrBlinkTime` 行为不变。

## 7. 第四阶段：协议层和业务层解耦

目标：`modbus.c` 和 `ags_mb.c` 不再直接修改业务全局状态。

### 7.1 新增 `app_service.h`

建议接口：

```c
typedef enum {
    APP_OK = 0,
    APP_ERR_INVALID_PARAM,
    APP_ERR_BUSY,
    APP_ERR_FORBIDDEN,
    APP_ERR_STORAGE,
} AppResult;

AppResult App_SetAddress(AppContext *ctx, uint8_t addr);
AppResult App_SetSpeed(AppContext *ctx, uint8_t speed);
AppResult App_SetBaudrate(AppContext *ctx, BaudRate_T baudrate);
AppResult App_SetProtocol(AppContext *ctx, Protocol_T protocol);
AppResult App_SetTargetPort(AppContext *ctx, uint8_t port);
AppResult App_SetGodMode(AppContext *ctx, GodMode_T mode);
AppResult App_SetChannelCount(AppContext *ctx, uint8_t count);
AppResult App_SetRdcRate(AppContext *ctx, uint8_t rate);
AppResult App_SetHalfSeal(AppContext *ctx, bool enabled);
AppResult App_ResetValve(AppContext *ctx);

uint8_t App_GetCurrentPort(const AppContext *ctx);
uint8_t App_GetTargetPort(const AppContext *ctx);
uint8_t App_GetValveState(const AppContext *ctx);
uint32_t App_GetMoveCount(const AppContext *ctx);
```

### 7.2 服务层职责

服务层负责：

- 参数范围校验。
- 判断当前状态是否允许执行操作。
- 更新 `AppContext`。
- 必要时写 EEPROM。
- 调用阀模块或硬件模块。

协议层负责：

- 解析帧。
- 检查功能码。
- 检查寄存器地址。
- 调用 `App_Service`。
- 组装响应。

### 7.3 迁移顺序

先迁移写操作，再迁移读操作。

优先迁移：

- 设置地址
- 设置速度
- 设置通道数
- 设置目标通道
- 设置波特率
- 设置协议类型
- 设置减速比
- 设置半通道
- 设置回复模式

原因：写操作副作用最大，最需要统一入口。

## 8. 第五阶段：统一寄存器映射

目标：AGS 和 Modbus 共用一套业务寄存器读写规则。

### 8.1 新增 `register_map.h`

```c
#ifndef REGISTER_MAP_H
#define REGISTER_MAP_H

#include "app_service.h"

AppResult Register_Read(uint16_t reg, uint16_t *value);
AppResult Register_Write(uint16_t reg, uint16_t value);

#endif
```

### 8.2 新增 `register_map.c`

职责：

- 管理寄存器地址到业务字段的映射。
- 调用 `App_Service`。
- 处理只读、只写、读写权限。
- 处理安全码、工厂模式、老化模式相关限制。

### 8.3 Modbus 迁移示例

迁移前：

```c
void mb_WriteHolding(uint16_t _regAddr, uint16_t _value)
{
    /* 大量 switch，直接改 syspara / valve / EEPROM */
}
```

迁移后：

```c
void mb_WriteHolding(uint16_t reg, uint16_t value)
{
    AppResult result = Register_Write(reg, value);
    if (result != APP_OK) {
        modbus.ErrorState = MB_ERROR_DATA;
        return;
    }

    MB_SET_HOLDING(reg, value);
}
```

### 8.4 验收标准

- `modbus.c` 不再直接写 `valve.xxx`、`syspara.xxx`、`rdc.xxx`。
- `ags_mb.c` 与 `modbus.c` 不再各自实现一套业务写入规则。
- 保持原有寄存器地址和响应格式。

## 9. 第六阶段：阀控制状态机化

目标：统一阀状态跳转入口，减少直接写 `valve.status`。

### 9.1 当前状态问题

当前状态定义：

```c
#define VALVE_NONE          0x00
#define VALVE_INITING       0x80
#define VALVE_RUN_END       0x40
#define VALVE_RUN_ERR       0x20
#define VALVE_RUNNING       0x08
```

问题：

- 既像 bit flag，又像 enum。
- 不同模块直接写 `valve.status`。
- 超时保护在 `main.c`，状态处理在 `motor.c`，错误闪烁又在 `main.c`。

### 9.2 新增状态类型

```c
typedef enum {
    VALVE_STATE_BOOT = 0,
    VALVE_STATE_INITING,
    VALVE_STATE_IDLE,
    VALVE_STATE_MOVING,
    VALVE_STATE_DONE,
    VALVE_STATE_ERROR,
    VALVE_STATE_AGING,
} ValveState;

typedef enum {
    VALVE_EVT_START_INIT,
    VALVE_EVT_MOVE_TO_PORT,
    VALVE_EVT_SENSOR_CHANGED,
    VALVE_EVT_TIMEOUT,
    VALVE_EVT_STOP,
    VALVE_EVT_RESET,
} ValveEvent;
```

### 9.3 过渡接口

先不要一次性重写 `bsp_ValveProcess()`，先建立语义函数：

```c
ValveState Valve_GetState(const AppContext *ctx);
void Valve_SetState(AppContext *ctx, ValveState state);
bool Valve_IsBusy(const AppContext *ctx);
bool Valve_IsError(const AppContext *ctx);
void Valve_SetError(AppContext *ctx);
```

逐步替换：

```c
valve.status == VALVE_RUNNING
valve.status & VALVE_INITING
valve.status = VALVE_RUN_ERR
```

### 9.4 超时保护迁移

当前 `task_Cycle_1Sec()` 中有阀超时保护。建议迁入：

```c
void Valve_CheckTimeout(AppContext *ctx);
```

`task_Cycle_1Sec()` 只保留调度调用：

```c
Valve_CheckTimeout(&g_app);
AppParam_FlushCountersIfNeeded(&g_app);
```

### 9.5 验收标准

- 直接写 `valve.status` 的位置显著减少。
- 错误状态统一通过 `Valve_SetError()` 设置。
- 阀超时逻辑从 `main.c` 移到阀模块。
- 老化模式行为保持不变。

## 10. 第七阶段：主循环瘦身

目标：让 `main()` 只做初始化编排和循环调度。

### 10.1 建议拆分

```c
static void System_Init(void);
static void Board_Init(void);
static void App_Init(void);
static void Protocol_Init(void);
static void App_RunOnce(void);
```

最终结构：

```c
int main(void)
{
    System_Init();
    Board_Init();
    App_Init();
    Protocol_Init();

    while (1) {
        App_RunOnce();
    }
}
```

### 10.2 `App_RunOnce()`

建议职责：

```c
static void App_RunOnce(void)
{
#ifndef AGING_MODE
    Valve_InitProcess(&g_app);
    Valve_Process(&g_app);
    Protocol_Poll();
    App_TaskScheduler(&g_app);
    App_DebugOut(&g_app);
#endif
    Valve_AgingProcess(&g_app);
    App_ErrorBlink(&g_app);
}
```

### 10.3 验收标准

- `main()` 行数显著减少。
- 启动顺序与原逻辑一致。
- 协议选择逻辑不变。
- 老化模式条件编译逻辑不变。

## 11. 第八阶段：硬件访问封装

目标：业务代码不直接操作 GPIO 宏。

### 11.1 新增 `board_io.h`

```c
void Board_Rs485SetRx(void);
void Board_Rs485SetTx(void);

void Board_ValveEnable(bool enable);
void Board_ValveReset(bool active);
void Board_ValveSetDir(bool dir);
void Board_ValvePulse(void);

void Board_LedSet(bool on);
void Board_LedToggle(void);

void Board_MotorSetCurrent(uint8_t current);
```

### 11.2 新增 `board_io.c`

短期内部仍使用旧宏：

```c
void Board_ValveEnable(bool enable)
{
    VALVE_ENA = enable ? ENABLE : DISABLE;
}
```

### 11.3 迁移对象

逐步替换：

- `VALVE_ENA = DISABLE`
- `LED_WORK = !LED_WORK`
- `RX_EN()`
- `TX_EN()`
- `ISET(valve.iSet)`

### 11.4 验收标准

- 业务模块不直接操作 RS485 收发方向宏。
- 阀控制模块通过 `Board_Valve*` 操作引脚。
- LED 闪烁通过 `Board_LedToggle()`。

## 12. 第九阶段：目录重组

目标：在逻辑边界稳定后，再移动文件。

### 12.1 第一轮只新增文件

不要立即移动老文件。先新增：

```text
SRC/APP/app_context.c
SRC/APP/app_param.c
SRC/APP/app_service.c
SRC/APP/register_map.c
SRC/HARDWARE/board_io.c
```

## 15. 风险点与规避方式

### 15.1 EEPROM 兼容风险

风险：

- 地址或长度变化会导致现场设备参数读取错误。

规避：

- 所有 `ADDR_*` 和 `LEN_*` 迁移后必须逐项比对。
- 不在重构提交中调整参数布局。

### 15.2 协议兼容风险

风险：

- AGS 或 Modbus 响应格式变化。
- CRC 高低字节顺序变化。
- 广播地址处理变化。

规避：

- 协议帧解析和响应组装先不优化。
- 只把业务读写动作迁到服务层。

### 15.3 启动顺序风险

风险：

- 时钟、delay、串口、IIC、定时器、GPIO 初始化顺序变化可能造成异常。

规避：

- `main()` 瘦身时保持原调用顺序。
- 只做函数提取，不调整顺序。

### 15.4 多硬件版本风险

风险：

- 901/906/909 GPIO 映射、细分、最大电流、方向开关不同。

规避：

- `board_config.h` 和 `board_pins.h` 迁移后逐 Target 编译。
- 不在同一提交中修改业务逻辑和硬件宏。

### 15.5 老化模式风险

风险：

- `AGING_MODE` 会改变主循环路径，容易漏测。

规避：

- 老化模式相关逻辑单独保留。
- 状态机迁移后单独验证 `bsp_ValveAgingMode()`。

## 16. 第一刀建议

最推荐先做：

```text
app_context.h/.c
app_param_map.h
app_param_defaults.h
app_param.h/.c
```

然后把 `main.c` 中的参数初始化链路迁出：

```text
param_Read()
param_Write()
ParameterInit()
```

原因：

- 切口清晰。
- 风险相对可控。
- 能立即减少 `main.c` 复杂度。
- 能为后续协议层和状态机重构提供稳定的参数访问入口。

第一阶段完成后的目标状态：

```text
main.c
  |-- 只负责启动顺序和主循环

app_param.c
  |-- 负责 EEPROM 参数读写
  |-- 负责默认值写入
  |-- 负责参数范围修复

app_context.c
  |-- 集中业务状态
```

## 17. 最终验收标准

重构完成后应达到：

- `common.h` 不再是全项目大总线。
- `main.h` 不再承载 EEPROM 地址和大量全局变量。
- `main.c` 只负责初始化编排和主循环。
- 参数读写逻辑集中在 `app_param.c`。
- 协议层不直接写业务结构体。
- 寄存器映射有统一入口。
- 阀状态跳转有统一入口。
- GPIO 操作被 `board_io` 封装。
- 绝大多数业务模块不再直接依赖所有头文件。
- 新增协议、修改寄存器、修改阀控制、适配硬件版本可以分别进行，不再牵一发动全身。
